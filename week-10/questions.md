# Вопросы для самопроверки

1. Что такое слой в Docker и почему порядок команд в Dockerfile важен для кэширования?

**Ответ:** Слой — это промежуточный образ, который создаётся после каждой инструкции Dockerfile (RUN, COPY, ADD). Docker кэширует слои и при повторной сборке использует кэш, если инструкция и её контекст не изменились. Порядок команд важен: часто изменяемые файлы (например, исходный код) нужно копировать в конце, чтобы кэш предыдущих слоёв (установка зависимостей) использовался эффективно.

2. В чем суть Multi-stage build? Какие проблемы он решает?

**Ответ:** Multi-stage build позволяет использовать несколько базовых образов в одном Dockerfile. На первом этапе (builder) устанавливаются все зависимости и компиляторы, а на финальном этапе копируется только результат сборки. Это решает проблемы: уменьшает размер образа (не нужны компиляторы в runtime), повышает безопасность (нет лишних инструментов), упрощает поддержку.

3. Зачем нужен файл `.dockerignore`?

**Ответ:** `.dockerignore` исключает файлы и директории из контекста сборки Docker. Это нужно чтобы: ускорить сборку (меньше файлов копируется), уменьшить размер образа (нет лишних файлов), повысить безопасность (не попадают секреты, ключи, .env файлы).

4. Чем отличаются инструкции `CMD` и `ENTRYPOINT`?

**Ответ:** `ENTRYPOINT` задаёт основную команду контейнера, которая всегда выполняется. `CMD` задаёт аргументы по умолчанию для ENTRYPOINT или команду, которую можно переопределить при запуске. При указании обоих, CMD становится аргументами для ENTRYPOINT. ENTRYPOINT сложнее переопределить (нужен флаг --entrypoint).

5. Почему рекомендуется использовать `COPY` вместо `ADD`?

**Ответ:** `COPY` выполняет только копирование файлов из контекста сборки в образ. `ADD` имеет дополнительную функциональность: может скачивать файлы по URL и автоматически распаковывать архивы. Эта неявная функциональность делает поведение ADD менее предсказуемым. COPY проще и прозрачнее, поэтому его рекомендуют использовать по умолчанию.

6. Почему приложение в контейнере должно слушать `0.0.0.0`, а не `127.0.0.1`?

**Ответ:** `127.0.0.1` (localhost) доступен только внутри контейнера. `0.0.0.0` означает «все интерфейсы» — приложение слушает на всех сетевых интерфейсах контейнера, включая тот, который проброшен наружу через `-p`. Если слушать только 127.0.0.1, то извне контейнера приложение будет недоступно.
