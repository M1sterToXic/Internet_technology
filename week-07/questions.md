# Вопросы для самопроверки

## 1. Зачем нужен файл `.proto`? Почему мы не пишем классы сообщений сразу на Python?

Файл `.proto` — это контракт (IDL — Interface Definition Language), который описывает структуру данных и сервисы.

**Преимущества:**
- **Языконезависимость** — один `.proto` файл генерирует код для Python, Java, Go, C++ и других языков
- **Валидация** — компилятор проверяет типы и структуру на этапе генерации
- **Документация** — контракт явно описывает API
- **Совместимость** — легче отслеживать изменения и версионность

## 2. Что такое Unary RPC? Чем он концептуально похож на REST запрос?

**Unary RPC** — это обычный вызов удалённой процедуры, где клиент отправляет один запрос и получает один ответ.

**Сходство с REST:**
- Один запрос → один ответ (Request/Response)
- Клиент ждёт ответа перед продолжением работы
- Похож на HTTP GET/POST запрос

**Пример:**
```protobuf
rpc GetOrder (GetOrderRequest) returns (GetOrderResponse);
```
Аналог в REST: `GET /orders/1`

## 3. Почему gRPC обычно эффективнее REST (JSON) по процессору и трафику?

**Бинарный формат Protobuf:**
- JSON: `{"id": "1", "name": "Order"}` — 35 байт (текст + кавычки + скобки)
- Protobuf: бинарное представление — ~15-20 байт

**Преимущества:**
- Меньше размер сообщений (нет имён полей в каждом сообщении)
- Быстрее сериализация/десериализация (бинарный парсинг vs текстовый)
- HTTP/2 мультиплексирование — одно соединение на все запросы

## 4. Что такое "обратная совместимость" в Protobuf и зачем нужны номера полей (тэги, например `= 1`)?

**Номера полей (тэги)** — это уникальные идентификаторы полей в бинарном формате.

**Обратная совместимость:**
- Можно добавлять новые поля с новыми номерами — старые клиенты их игнорируют
- Нельзя менять номера существующих полей — сломается совместимость
- Можно удалять поля, но номер должен зарезервироваться (`reserved 3;`)

**Пример:**
```protobuf
message Order {
  string id = 1;      // нельзя менять номер!
  string name = 2;    // нельзя менять номер!
  int32 priority = 3; // новое поле - OK
}
```

## 5. Что такое Stub (стаб) на клиенте и Servicer на сервере?

**Stub (клиентский):**
- Генерируется автоматически из `.proto`
- Предоставляет методы для вызова RPC
- Пример: `stub = OrdersServiceStub(channel)`

**Servicer (серверный):**
- Базовый класс, который вы наследуете
- Реализуете методы сервиса
- Пример: `class OrdersServiceServicer(OrdersServiceServicer):`

## 6. В каких случаях вы бы выбрали REST, а в каких — gRPC?

**REST (JSON/HTTP):**
- Публичные API для внешних разработчиков
- Простые CRUD приложения
- Когда важна читаемость и отладка
- Браузерные клиенты (без gRPC-Web)

**gRPC:**
- Микросервисы внутри компании (service-to-service)
- Высокие требования к производительности
- Streaming (real-time данные, видео, логи)
- Мобильные приложения (экономия трафика)
- Строгая типизация и контракт
