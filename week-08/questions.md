# Вопросы для самопроверки

## 1. Что такое Server Streaming? Приведите пример использования.

**Server Streaming** — это тип gRPC вызова, при котором клиент отправляет один запрос, а сервер отвечает потоком сообщений. Соединение остаётся открытым до тех пор, пока сервер не отправит все данные.

**Примеры использования:**
- Лента новостей или биржевых котировок
- Стриминг логов в реальном времени
- Загрузка больших файлов с прогрессом
- Real-time уведомления (например, статусы заказов)

## 2. Чем Server Streaming отличается от просто получения большого списка (List) в одном ответе?

| Server Streaming | List в одном ответе |
|-----------------|---------------------|
| Данные приходят порциями по мере готовности | Клиент ждёт весь ответ целиком |
| Можно начинать обработку сразу | Обработка только после получения всего ответа |
| Эффективно для больших данных | Риск таймаута при больших объёмах |
| Соединение держится открытым | Соединение закрывается после ответа |

## 3. Какие метрики производительности существуют (Latency, Throughput, CPU Usage)?

- **Latency (Задержка)** — время от отправки запроса до получения ответа. Измеряется в миллисекундах.
- **Throughput (Пропускная способность)** — количество запросов, обрабатываемых в секунду (RPS — requests per second).
- **CPU Usage** — загрузка процессора при обработке запросов.
- **Memory Usage** — потребление памяти.
- **Error Rate** — процент ошибочных запросов.

## 4. Почему важно переиспользовать канал (Channel) в gRPC клиенте при замерах производительности?

Создание нового канала (Channel) для каждого запроса — дорогая операция:
- Требуется установка нового TCP-соединения
- Происходит handshake HTTP/2
- Выделяются ресурсы

При переиспользовании одного канала:
- Все запросы идут через одно TCP-соединение (мультиплексирование HTTP/2)
- Снижаются накладные расходы
- Замеры отражают реальную производительность gRPC, а не overhead на создание соединений

## 5. В каких сценариях REST может оказаться быстрее или удобнее gRPC?

**REST удобнее:**
- Простые CRUD-операции с небольшими данными
- Когда важна простота отладки (JSON читается легче бинарного формата)
- Для публичных API (не требуется генерация кода)
- Когда клиенты не могут использовать gRPC (браузеры без gRPC-Web)
- Быстрое прототипирование

**REST может быть быстрее:**
- При кэшировании на уровне HTTP (CDN, браузер)
- Для очень простых запросов с маленьким JSON

## 6. Что такое "Head-of-Line Blocking" и как HTTP/2 (и gRPC) решает эту проблему?

**Head-of-Line Blocking (HOL)** — проблема HTTP/1.1, когда следующий запрос не может быть обработан, пока не завершится предыдущий, даже если они независимы.

**Как HTTP/2 решает:**
- **Мультиплексирование** — несколько запросов/ответов передаются одновременно в рамках одного соединения
- **Фреймы** — данные разбиваются на фреймы, которые могут идти в любом порядке
- **Потоки** — каждый запрос имеет свой ID, сервер знает какой фрейм к какому запросу относится

gRPC поверх HTTP/2 наследует эти преимущества и дополнительно использует бинарный формат Protobuf для эффективной передачи данных.
