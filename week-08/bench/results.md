# Результаты замеров

## Проект: orders-s13

## Методология
- Количество запросов: 1000
- REST эндпоинт: GET http://localhost:8222/api/orders/1
- gRPC метод: GetOrder (Unary)
- Измерения проводились на localhost для исключения сетевых задержек

## Результаты

| Метрика | REST | gRPC |
|---------|------|------|
| Время (1000 запросов) | ~2.5 сек | ~0.8 сек |
| RPS (запросов/сек) | ~400 | ~1250 |
| Latency (средняя) | ~2.5 мс | ~0.8 мс |

## Выводы

gRPC показал значительно лучшую производительность по сравнению с REST:
- **gRPC быстрее в ~3 раза** по сравнению с REST при одинаковом количестве запросов
- Основная причина производительности gRPC — бинарный формат Protocol Buffers, который требует меньше времени на сериализацию/десериализацию по сравнению с JSON
- HTTP/2 использует одно TCP-соединение для всех запросов (мультиплексирование), что снижает накладные расходы на установку соединений
- REST проще в отладке и поддержке, но gRPC выигрывает в производительности для высоконагруженных систем

## Streaming метод

Реализован Server Streaming метод `StreamOrders`, который возвращает поток заказов. Это позволяет:
- Получать данные по мере их готовности без ожидания полного ответа
- Эффективно передавать большие объёмы данных
- Реализовывать real-time обновления (например, ленту заказов)
