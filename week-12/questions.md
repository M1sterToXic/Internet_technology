# Вопросы для самопроверки

1. Почему мы создаем Deployment, а не сразу Pod?

Deployment управляет жизненным циклом подов: обеспечивает нужное количество реплик, автоматически перезапускает упавшие поды и поддерживает rolling update. Одиночный Pod при падении просто исчезает без автоматического восстановления.

2. В чем разница между Liveness и Readiness пробами? Что произойдет, если провалится каждая из них?

Liveness проверяет, живо ли приложение. Если probe провален — kubelet убивает и пересоздаёт контейнер. Readiness проверяет, готов ли pod принимать трафик. Если провалена — pod временно исключается из балансировки, но не перезапускается.

3. Что такое "requests" и "limits" для CPU и памяти? Почему важно их указывать?

Requests — минимальные ресурсы, которые гарантированно выделяются поду. Limits — максимальные, выше которых контейнер будет ограничен (CPU throttling) или убит (OOM Kill). Без них «прожорливый» контейнер может исчерпать ресурсы ноды и нарушить работу соседних подов.

4. Как Service находит свои Pod-ы? (Подсказка: Labels & Selectors).

Service использует selector с набором label'ов и направляет трафик на все pod'ы, у которых совпадают эти label'ы. Kubernetes автоматически обновляетEndpoints при создании или удалении подов.

5. Какой тип сервиса (ClusterIP, NodePort, LoadBalancer) используется для внутреннего общения, а какой — для внешнего доступа?

ClusterIP — только для внутреннего общения внутри кластера. NodePort открывает порт на каждой ноде для внешнего доступа. LoadBalancer создаёт внешний балансировщик через облачный провайдер.

6. Что такое "Rolling Update" в контексте Деплоймента?

Это стратегия обновления, при которой Kubernetes постепенно заменяет старые поды новыми, не прерывая сервис: создаёт новый pod, ждёт его readiness, затем удаляет старый. Пользователи не замечают простоя.
