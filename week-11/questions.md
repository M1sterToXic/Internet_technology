# Вопросы для самопроверки

1. Зачем создавать пользовательскую сеть (bridge network) в Docker Compose, если есть дефолтная?

В дефолтной сети сервисы тоже могут общаться по именам, но пользовательская сеть даёт изоляцию и контроль. В ней работает встроенный DNS, можно задавать свои алиасы, ограничивать видимость сервисов и настраивать параметры (MTU, подсеть). Кроме того, при использовании нескольких compose-проектов дефолтные сети могут конфликтоваться, а именованные — нет.

2. В чем разница между `build` и `image` в описании сервиса?

`build` указывает путь к Dockerfile и собирает образ локально. `image` — использует готовый образ из реестра (например, Docker Hub). Можно использовать оба: `build` собирает образ и автоматически тегирует его именем из `image`.

3. Что делает `depends_on`? Гарантирует ли он, что приложение в зависимом контейнере полностью инициализировалось?

Сам по себе `depends_on` лишь задаёт порядок создания контейнеров. Он не ждёт, пока приложение внутри контейнера начнёт принимать соединения. Чтобы дождаться реальной готовности, нужно использовать `depends_on` с `condition: service_healthy` вместе с `healthcheck`.

4. Как работают healthcheck-и и как использовать их в `depends_on`?

Healthcheck — это команда, которую Docker периодически выполняет внутри контейнера. Параметры `interval`, `timeout`, `retries` определяют частоту и пороги. Если healthcheck проходит, контейнер получает статус `healthy`. В `depends_on` можно указать `condition: service_healthy`, и тогда зависимый сервис начнёт запуск только после того, как основной сервис станет здоровым.

5. Как передать переменные окружения в контейнер через Compose (минимум 2 способа)?

— Напрямую в YAML через `environment: KEY=value`.
— Через внешний файл `.env` с помощью `env_file: .env`.
— Также можно использовать `environment: - KEY=${VAR}` для подстановки из окружения хоста.

6. Почему Docker Compose редко используют в продакшене (хотя это возможно)?

Compose не обеспечивает оркестration-возможности: автоматический restart при падении, горизонтальное масштабирование, zero-downtime деплой, распределение по узлам кластера, централизованное управление конфигурацией и секретами. Для продакшена используют Kubernetes, Docker Swarm или ECS, которые решают эти задачи.
